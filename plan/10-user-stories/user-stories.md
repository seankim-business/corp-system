# User Stories: Company OS + Multi-Agent System

## 개요

이 문서는 사용자 관점에서 Company OS를 어떻게 경험하는지 설명합니다.
모든 상호작용은 **Slack**을 통해 이루어지며, 복잡한 시스템은 뒤에 숨어있습니다.

---

## 핵심 원칙

### 유저가 경험하는 것
- **단일 인터페이스**: Slack만 사용
- **자연어 대화**: 명령어 암기 불필요
- **투명한 진행**: 에이전트 작업 과정 실시간 확인
- **Human-in-the-Loop**: 중요 결정은 반드시 사람이 승인

### 유저가 신경 쓰지 않아도 되는 것
- 문서가 어디에 저장되어 있는지 (Notion? Drive? GitHub?)
- 어떤 에이전트가 어떤 일을 하는지
- 시스템 간 동기화
- API나 기술적 연결

## User Story 목록

### US-001: 일일 브리핑 받기

**As a** 팀원
**I want to** 아침에 오늘 할 일과 대기 중인 요청을 한눈에 보고 싶다
**So that** 하루를 효율적으로 시작할 수 있다

#### 시나리오

```
🤖 [secretary] 좋은 아침입니다, 김팀장님!

📅 오늘 일정:
├─ 10:00 마케팅 캠페인 검토 미팅
├─ 14:00 Q2 OKR 체크인
└─ 16:30 1:1 with 박대리

📌 대기 중인 승인 요청 (2건):
├─ [campaign-agent] 신규 캠페인 브리프 최종 검토
└─ [content-agent] 블로그 포스트 발행 승인

⚡ 긴급 알림:
└─ [sales-agent] A사 계약 만료 D-7, 갱신 논의 필요

오늘도 화이팅입니다! 🚀
```

#### 수용 조건 (Acceptance Criteria)
- [ ] 매일 아침 9시에 자동 브리핑 발송
- [ ] 개인 캘린더 연동 (Google Calendar)
- [ ] 대기 중인 승인 요청 집계
- [ ] 긴급도에 따른 우선순위 표시
- [ ] 사용자별 커스터마이징 가능 (브리핑 시간, 포함 항목)

### US-002: 업무 요청하기

**As a** 팀원
**I want to** 자연어로 업무를 요청하고 싶다
**So that** 복잡한 명령어나 도구 없이 일을 진행할 수 있다

#### 시나리오

**유저 요청:**
```
@agent 다음 주 제품 런칭 마케팅 계획 세워줘
```

**시스템 응답:**
```
🤖 [orchestrator] 요청을 분석했습니다.

   📋 태스크: 제품 런칭 마케팅 계획 수립
   👥 담당 에이전트:
   ├─ campaign-agent (리드)
   ├─ content-agent
   └─ design-agent

   ⏱️ 예상 소요: 약 30분

   시작할까요? [✅ 시작] [📝 요구사항 추가]
```

**유저 확인 후:**
```
🤖 [campaign-agent] 마케팅 계획 수립을 시작합니다.

   📄 관련 SOP 로드: /sops/marketing/product-launch.md
   📊 과거 런칭 데이터 분석 중...

   [■■■□□□□□□□] 30% - Value Chain 분석 중
```

#### 수용 조건
- [ ] 자연어 요청 파싱 및 의도 분석
- [ ] 적절한 에이전트 자동 라우팅
- [ ] 복잡한 요청은 여러 에이전트 협업 구성
- [ ] 시작 전 유저 확인 (선택적)
- [ ] 실시간 진행률 표시

```
   1. "더 빠른 워크플로우"
   2. "팀 협업의 새로운 기준"
   3. "AI가 함께하는 업무"
```

#### 수용 조건
- [ ] 에이전트 간 메시지 교환 실시간 표시
- [ ] 각 에이전트의 현재 상태 표시
- [ ] 중간 산출물 미리보기
- [ ] 유저가 중간에 피드백/개입 가능
- [ ] 협업 흐름 다이어그램 (선택적)

### US-004: 중요 결정 승인하기

**As a** 의사결정권자
**I want to** 중요한 액션 전에 승인 요청을 받고 싶다
**So that** 실수나 오류를 사전에 방지할 수 있다

#### 시나리오

**승인 요청:**
```
🤖 [campaign-agent]
   ⚠️ 승인 요청 #127

   📋 내용: 마케팅 예산 $50,000 집행 계획

   ┌─────────────────────────────────┐
   │ 항목             │ 금액        │
   ├─────────────────────────────────┤
   │ 페이스북 광고    │ $20,000     │
   │ 인플루언서 협찬  │ $15,000     │
   │ 런칭 이벤트      │ $15,000     │
   └─────────────────────────────────┘

   📎 상세 계획서: [Google Drive 링크]

   [✅ 승인] [❌ 거절] [💬 수정 요청] [⏸️ 보류]
```

**수정 요청 시:**
```
유저: 인플루언서 예산을 줄이고 이벤트에 더 투자해줘
```

**에이전트 응답:**
```
🤖 [campaign-agent]
   ✅ 수정 반영 완료

   변경 내역:
   ├─ 인플루언서: $15,000 → $5,000 (↓$10,000)
   └─ 런칭 이벤트: $15,000 → $25,000 (↑$10,000)

   [✅ 승인] [❌ 거절] [💬 추가 수정]
```

#### 승인이 필요한 액션 유형

| 카테고리 | 액션 예시 | 승인 권한 |
|---------|----------|----------|
| 예산 | 일정 금액 이상 지출 | 팀장/CFO |
| 외부 커뮤니케이션 | 고객 이메일, 보도자료 | 팀장/PR팀 |
| 계약 | 계약서 발송, 서명 요청 | 법무/대표 |
| 코드 배포 | 프로덕션 배포 | Tech Lead |
| 인사 | 채용 오퍼, 평가 | HR/팀장 |
| 데이터 | 민감 데이터 접근/수정 | 데이터 오너 |

#### 수용 조건
- [ ] 승인 필요 액션 자동 감지
- [ ] 적절한 승인권자에게 알림
- [ ] 버튼 또는 자연어로 응답 가능
- [ ] 승인 이력 기록 (감사 추적)
- [ ] 승인 대기 시간 초과 알림
- [ ] 위임 승인 지원 (부재 시)

### US-009: 권한에 따른 접근 제어

**As a** 시스템 관리자
**I want to** 역할에 따라 정보와 기능에 대한 접근을 제어하고 싶다
**So that** 민감 정보를 보호하고 권한을 명확히 할 수 있다

#### 시나리오: 권한 없는 요청

```
일반 직원: @agent 전체 급여 데이터 보여줘

🤖 [hr-agent]
   ⚠️ 접근 권한이 없습니다.

   급여 정보 조회 권한:
   ├─ HR팀 (전체)
   ├─ 경영진 (전체)
   └─ 일반 직원 (본인만)

   ℹ️ 본인 급여 정보는 조회 가능합니다.
   "내 급여 정보 보여줘"로 요청해주세요.

   권한이 필요하시면 HR팀에 요청해주세요.
```


- Slack 장애 시 대체 채널 (이메일)
